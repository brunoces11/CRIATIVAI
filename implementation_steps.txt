1. OBJETIVO E REGRA DE USO DESTE ARQUIVO
===============================================================================

Este arquivo é o guia operacional para concluir o escopo do plano anexado sem
reimplementar a fundação que já existe, sem alterar a landing page fora do
necessário e sem introduzir arquitetura desnecessária.

A execução deve:

- preservar React + Vite + TypeScript no frontend;
- preservar FastAPI + Python no runtime;
- preservar SQLite + SQLAlchemy 2 + Alembic;
- usar um único container e um único processo Uvicorn com um worker;
- usar a OpenAI somente no backend;
- usar a Responses API do SDK oficial da OpenAI, com streaming e tool calling;
- usar Google OAuth e Google Calendar somente no backend;
- manter o histórico completo no SQLite;
- usar o Traefik para proteger /adm e /api/admin/* em produção;
- manter Node apenas no desenvolvimento e no estágio de build;
- não adicionar Next, Vinext, Drizzle, D1, Workers, Redis, PostgreSQL, Nginx
  interno, filas, microsserviços, RAG, embeddings ou banco vetorial;
- modificar somente o necessário para a etapa em andamento;
- iniciar automaticamente a etapa seguinte somente após a etapa atual passar
  por testes, correções, commit e push;
- pausar quando um gate de decisão humana deste documento ainda não estiver
  resolvido.

As estimativas são timeboxes para distribuir bem a capacidade de execução.
Elas não autorizam cortar testes, segurança ou critérios de aceite para cumprir
um relógio.

2.1. Fundação que já existe e deve ser reaproveitada

- Frontend React 19, Vite 8 e TypeScript estrito.
- Roteamento SPA simples em src/App.tsx, sem biblioteca de router.
- Três páginas públicas existentes: /, /human-resources e /style.

- Backend mínimo em backend/app.
- GET /api/health.
- POST /api/chat com streaming NDJSON fake.
- GET /api/conversations/current.
- FastAPI servindo dist/, com fallback de SPA que não captura /api/*.
- Cache longo para assets com hash e no-store para index.html.
- SQLAlchemy síncrono, foreign keys e WAL no SQLite.
- Alembic configurado e migration inicial já versionada.
- Tabelas iniciais conversations, messages, bookings e oauth_states.
- Dockerfile.deploy multi-stage com Node no build e Python no runtime.
- Teste vertical que sobe FastAPI real, aplica migration e valida persistência.
- .env real ignorado pelo Git e .dockerignore excluindo segredos e dados.

Não recriar essa fundação. Evoluir os arquivos atuais e só separar módulos
quando a responsabilidade nova justificar a separação.

2.2. Lacunas e conflitos identificados

- Não existe Chat UI.
- Não existe frontend consumindo o streaming.
- O backend ainda responde com texto fake.
- O SDK da OpenAI e as bibliotecas Google ainda não estão instalados.
- O Settings atual declara apenas banco e dist.
- O Settings atual rejeita variáveis presentes no .env.
- Não existem prompt, FAQ e base de conhecimento do SDR.
- Não existem rate limiting, idempotência de turnos, logs seguros ou timeout
  explícito.
- O session_id tem alta entropia na criação, mas a validação e o comportamento
  para ID inválido/desconhecido ainda precisam ser endurecidos.
- O schema inicial não possui todos os campos necessários para status de
  mensagem, última atividade, reagendamento e estado de booking.
- Não existem APIs ou interface administrativas.
- Não existem OAuth, FreeBusy ou mutações do Google Calendar.
- deploy.env ainda classifica o projeto como static, sem runtime Python, porta
  8000, volume persistente ou arquivo de ambiente do runtime.
- README ainda descreve deploy estático.
- .env.example está incompleto.
- Existe um segundo arquivo .env.exemple, com nome incorreto, sem referência na
  codebase. Ele deve ser consolidado no arquivo canônico após nova verificação.
- Existe .openai/hosting.json, que representa um alvo de hosting diferente do
  VPS Docker + Traefik definido como arquitetura final. Esse conflito deve ser
  resolvido antes do deploy, sem manter duas fontes de verdade concorrentes.

3. VIABILIDADE, JANELAS DE EXECUÇÃO E CONTINUIDADE AUTOMÁTICA
===============================================================================

O escopo final completo não deve ser prometido dentro de uma única janela de
6 horas. Há trabalho suficiente para dois blocos e existem ações externas que
não podem ser automatizadas com segurança:

- rotação de credenciais;
- fornecimento do prompt, FAQ e regras comerciais;
- consentimento OAuth na conta Google do proprietário;
- criação do usuário/hash administrativo;
- confirmação do hash imutável do plano de deploy;
- validação de um evento real e de um convite real no Calendar.

Distribuição recomendada:

BLOCO A - até 6 horas ativas

- Etapa 1: configuração segura e baseline recuperado;
- Etapa 2: Chat UI base;
- Etapa 3: integração vertical frontend/backend;
- Etapa 4: OpenAI real;
- Etapa 5: persistência, contexto e segurança mínima;
- Etapa 6: admin somente leitura;
- Etapa 7: runtime Docker candidato e regressão do bloco.

Entrega do Bloco A:

- chat real com streaming;
- restauração de sessão;
- histórico persistido e isolado;
- síntese comercial;
- admin de leitura;
- runtime FastAPI validado em container;
- sem Calendar ainda;
- sem deploy público ainda.

BLOCO B - aproximadamente 6 a 7 horas ativas, além das esperas humanas

- Etapa 8: Google OAuth;
- Etapa 9: disponibilidade e fusos;
- Etapa 10: criação de evento;
- Etapa 11: reagendamento e cancelamento;
- Etapa 12: tool loop completo e fluxo de agendamento no chat;
- Etapa 13: admin Calendar e proteção Traefik;
- Etapa 14: deploy transacional;
- Etapa 15: aceite final, limpeza e documentação.

Continuidade:

- Dentro de cada bloco, iniciar automaticamente a etapa seguinte quando todos
  os critérios da etapa atual estiverem verdes e o push estiver confirmado.
- Se um gate estiver pendente, pausar antes da primeira alteração dependente
  dele, registrar exatamente o que falta e pedir a decisão ao usuário.
- Não usar placeholders silenciosos em produção.
- Um placeholder explicitamente marcado só pode existir em desenvolvimento e
  o startup de produção deve falhar de forma segura se conteúdo obrigatório
  ainda estiver ausente.
- Se uma etapa ultrapassar 75 minutos por causa de uma descoberta não prevista,
  não ampliar silenciosamente o escopo. Dividir em subetapas A/B, mantendo os
  mesmos critérios de aceite e um commit/push para cada subetapa concluída.

5. PROTOCOLO OBRIGATÓRIO PARA TODAS AS ETAPAS
===============================================================================

Antes de modificar:

1. Confirmar que a etapa anterior foi enviada ao remoto.
2. Verificar branch, HEAD e git status.
3. Não sobrescrever mudanças locais do usuário.
4. Inspecionar apenas os arquivos envolvidos na etapa.
5. Confirmar que todos os gates da etapa estão resolvidos.
6. Registrar no resumo da etapa o escopo exato e os critérios de aceite.

Durante a implementação:

1. Reutilizar componentes, settings, sessão SQLAlchemy, schemas e helpers
   existentes.
2. Evitar classes abstratas, repository pattern, unit of work customizada e
   arquivos de função única sem justificativa.
3. Não mover a estrutura atual só por preferência arquitetural.
4. Não alterar conteúdo ou layout da landing page fora dos pontos necessários
   para anexar o widget.
5. Nunca imprimir prompts completos, mensagens, tokens, chaves ou credenciais.
6. Não executar chamadas reais em toda a suíte; usar mocks para OpenAI, Google
   Calendar e OAuth.

Fechamento obrigatório da etapa:

1. Executar primeiro os testes específicos da etapa.
2. Executar a regressão definida na própria etapa.
3. Corrigir todo erro introduzido ou revelado no escopo da etapa.
4. Reexecutar testes até todos passarem.
5. Executar git diff --check.
6. Revisar o diff e confirmar ausência de alteração fora do escopo.
7. Confirmar, sem exibir conteúdo, que staging não contém:
   - .env real;
   - banco SQLite;
   - arquivos WAL/SHM;
   - google-token.json;
   - chaves ou tokens;
   - credenciais Basic Auth;
   - conteúdo de /app/data ou data/.
8. Atualizar neste arquivo o status e as evidências da etapa, quando a execução
   real começar.
9. Criar um único commit descritivo da etapa.
10. Fazer push para origin da branch atual.
11. Confirmar que o commit remoto corresponde ao commit local.
12. Não usar force push.
13. Iniciar automaticamente a próxima etapa se não houver gate pendente.

Se o push falhar:

- não iniciar a etapa seguinte;
- diagnosticar divergência ou autenticação;
- nunca usar force push como atalho;
- pedir decisão do usuário se houver conflito remoto.
